# Contributing to IQToolkit

We welcome contributions to the IQToolkit ecosystem! This guide covers how to contribute to existing tools, create new tools, and improve the documentation hub.

## Ways to Contribute

### 1. Contribute to Existing Tools
- **Bug fixes** and **feature improvements**
- **Documentation** updates and examples
- **Testing** and bug reporting
- **Community support** in discussions and issues

### 2. Create New Tools
- **Build new tools** that fit the IQToolkit philosophy
- **Follow our guidelines** for consistency
- **Add comprehensive documentation**
- **Submit for inclusion** in the ecosystem

### 3. Improve the Documentation Hub
- **Fix documentation errors** or outdated information
- **Add examples** and tutorials
- **Improve the structure** and navigation
- **Suggest new content** or sections

## Tool Development Guidelines

### Philosophy
IQToolkit tools should be:
- **Focused**: Each tool solves a specific problem well
- **Independent**: Can be used standalone
- **Well-documented**: Clear installation and usage instructions
- **Open source**: MIT or similar permissive license

### Technical Requirements
- **Python 3.11+** for Python tools
- **Clear CLI interface** when applicable
- **Configuration support** (files, environment variables, CLI args)
- **Error handling** and helpful error messages
- **Tests** and CI/CD setup

### Documentation Structure
Your tool should include a `/docs` folder with:
- `index.md` - Tool overview and introduction
- `getting-started.md` - Installation and basic usage
- `configuration.md` - Configuration options
- `examples.md` - Usage examples
- `faq.md` - Frequently asked questions

## Documentation Sync Process

Once your tool is ready:

1. **Create documentation** in your tool's `/docs` folder
2. **Open a pull request** to add your tool to the sync configuration in this repository
3. **Documentation will be automatically synced** and published to this site
4. **Your tool appears** in the tools catalog

## Community Guidelines

- **Be respectful** and inclusive in all interactions
- **Use clear, descriptive commit messages**
- **Follow the existing code style** in each repository
- **Add tests** for new features and bug fixes
- **Update documentation** for any changes

## Getting Started

1. **Explore existing tools** to understand the patterns
2. **Join our discussions** at [GitHub Discussions](https://github.com/orgs/iqtoolkit/discussions)
3. **Start small** with bug fixes or documentation improvements
4. **Reach out** if you have questions or ideas

## Development Workflow

### For Tool Contributions
1. Fork the specific tool's repository
2. Create a feature branch
3. Make your changes with tests
4. Submit a pull request to the tool's repository

### For Documentation Hub
1. Fork this documentation repository
2. Make your improvements
3. Test locally with `make dev`
4. Submit a pull request

## Recognition

Contributors are recognized in:
- **Tool repositories** - Individual contributor lists
- **Release notes** - Acknowledgment of contributions
- **Community highlights** - Featured contributions in discussions

## Need Help?

- **General questions**: [GitHub Discussions](https://github.com/orgs/iqtoolkit/discussions)
- **Specific tool issues**: Visit the tool's GitHub repository
- **Documentation issues**: [Create an issue](https://github.com/iqtoolkit/docs.iqtoolkit.ai/issues)
